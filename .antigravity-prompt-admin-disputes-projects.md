## Part 1 — Disputes (`app/admin/disputes/page.tsx`)

Backend is fully built already, only frontend wiring is missing:
- `GET /disputes` — list all disputes, `@Roles(Role.ADMIN)` (`backend/src/modules/disputes/disputes.controller.ts:16`). Returns disputes with `contract.project.title` included. Service: `disputes.service.ts:10`.
- `GET /disputes/:id` — single dispute detail (`disputes.controller.ts:23`).
- `POST /disputes/:id/resolve` — resolve it, `@Roles(Role.ADMIN)` (`disputes.controller.ts:32`). Body shape (`backend/src/modules/disputes/dto/dispute.dto.ts`):
  ```ts
  { resolution: string; outcome: 'RELEASE' | 'REFUND' | 'SPLIT' }
  ```
  - `RELEASE` pays the student the full escrowed amount.
  - `REFUND` returns the full amount to the client and cancels the contract.
  - `SPLIT` splits the escrowed amount 50/50 between student and client.
  This single endpoint atomically updates wallets, escrow, contract/project status, and writes an audit log — the frontend just needs to collect `resolution` (free text) and `outcome` (radio/select) and POST it.

Replace the hardcoded empty state in `app/admin/disputes/page.tsx` with:
1. On mount, `apiClient('/disputes')` and render a list: project title, dispute status (`DisputeStatus` enum — check `backend/prisma/schema.prisma` for values), amount in escrow if available, created date.
2. For disputes with status not yet `RESOLVED`, show a "Resolve" action (button opening a dialog, consistent with other admin/client dialogs in this codebase — see `app/client/projects/[id]/page.tsx` for an existing Dialog pattern) that collects `resolution` text + `outcome` radio buttons, then `POST`s to `/disputes/:id/resolve` and refetches the list on success.
3. Keep the empty state for when the list is actually empty (zero disputes) — that's a legitimate state, not a stub.
4. Use the existing `useRequireAuth`, `apiClient`, `Card`/`Button`/`Dialog` components and styling conventions already used elsewhere in `app/admin/users/page.tsx` for visual consistency.

**Also fix the broken flow**: `app/dashboard/admin/page.tsx` has a "Review" button next to open disputes that currently links to `/admin/disputes` with no way to jump to the specific dispute. Update it to link to `/admin/disputes?id=<disputeId>` (or similar) and have the disputes page auto-open that dispute's resolve dialog if the query param is present — check `app/dashboard/admin/page.tsx` for the exact current link/button to confirm what needs changing.

## Part 2 — Projects (`app/admin/projects/page.tsx`)

No admin-scoped backend endpoint exists yet for this — only `GET /projects/my/list` (client-scoped, `Role.CLIENT`) and the public `GET /projects` (marketplace, filtered to `PUBLISHED` only) exist. Build:

### Backend (`backend/src/modules/admin/`)
Add to `admin.controller.ts` (already has `@Controller('admin')` + `@Roles(Role.ADMIN)` at the class level, so no extra guard needed per-route — follow the existing pattern used by `getUsers`):

```ts
@Get('projects')
@ApiOperation({ summary: 'Get paginated projects across all clients [ADMIN]' })
getProjects(
  @Query('page') page = 1,
  @Query('limit') limit = 20,
  @Query('status') status?: string,
  @Query('search') search?: string,
) {
  return this.adminService.getProjects(Number(page), Number(limit), status, search);
}

@Patch('projects/:id')
@ApiOperation({ summary: 'Admin override: cancel or archive any project [ADMIN]' })
adminUpdateProject(
  @CurrentUser() user: JwtPayload,
  @Param('id', ParseUUIDPipe) id: string,
  @Body() data: { status: 'CANCELLED' | 'ARCHIVED' },
) {
  return this.adminService.adminUpdateProject(user.sub, id, data.status);
}
```

In `admin.service.ts`, follow the exact pagination pattern already used by `getUsers` (`admin.service.ts:32`, `Promise.all([findMany, count])`, `skip`/`take`, `orderBy: createdAt desc`). For `getProjects`:
- `where` should support optional `status` filter and `search` (title/description, case-insensitive `contains`, same as `projects.service.ts`'s `findAll()` marketplace search at `backend/src/modules/projects/projects.service.ts:98`).
- `include`: `client: { select: { companyName: true, user: { select: { name: true } } } }`, `_count: { select: { applications: true } }`.
- Return `{ projects, pagination: { total, page, limit, totalPages } }` (same shape as `projects.service.ts findAll()` for frontend consistency).

For `adminUpdateProject`: look up the project (404 if missing), update its `status` directly via `this.prisma.project.update`, and write an `auditLog` entry (`action: 'PROJECT_CANCELLED_BY_ADMIN'` or `'PROJECT_ARCHIVED_BY_ADMIN'`, `resource: 'Project'`, `resourceId: id`) — follow the exact audit log pattern used in `banUser`/`shadowBanUser` in the same file. No need to validate current status transitions as strictly as `projects.service.ts`'s `update()` does — admin override should work regardless of current state (except maybe block on already-`CANCELLED`/`ARCHIVED`, return a `BadRequestException` if so).
