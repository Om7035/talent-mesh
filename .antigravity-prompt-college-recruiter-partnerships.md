# Task: College-Recruiter Partnership System

Build the feature as proposed: TPOs and Recruiters can form direct partnerships
(LinkedIn-connection-style), and once partnered, a TPO can push a student's profile
directly to a recruiter outside of any specific job posting.

Two product decisions are already made — implement to these, don't re-ask:
1. **Either side can initiate a partnership request** (TPO → Recruiter or Recruiter →
   TPO/College), not TPO-only.
2. **Silent push**: when a TPO pushes a student to a recruiter, the student is NOT
   notified. Only the recruiter gets a notification. (Revisit later if the product wants
   more transparency, but ship silent-only for now.)

## 1. Schema (`backend/prisma/schema.prisma`)

Add two new models. Match this repo's existing conventions (UUID `id @default(uuid())`,
`createdAt DateTime @default(now())`, `@@map("snake_case_table_name")`, explicit
`@relation` with named FKs — look at how `TpoRecommendation` or `Dispute` are modeled in
the same file for the exact style to copy).

```prisma
enum PartnershipStatus {
  PENDING
  ACTIVE
  REJECTED
}

enum PartnershipInitiator {
  TPO
  RECRUITER
}

model CollegePartnership {
  id            String                @id @default(uuid())
  collegeId     String
  college       College               @relation(fields: [collegeId], references: [id], onDelete: Cascade)
  recruiterId   String
  recruiter     Recruiter             @relation(fields: [recruiterId], references: [id], onDelete: Cascade)
  status        PartnershipStatus     @default(PENDING)
  initiatedBy   PartnershipInitiator
  initiatedById String                // userId of whoever sent the request, for audit/display
  respondedAt   DateTime?
  createdAt     DateTime              @default(now())
  updatedAt     DateTime              @updatedAt

  pushes        DirectTalentPush[]

  @@unique([collegeId, recruiterId])  // one partnership record per college-recruiter pair
  @@index([recruiterId, status])
  @@index([collegeId, status])
  @@map("college_partnerships")
}

model DirectTalentPush {
  id             String              @id @default(uuid())
  partnershipId  String
  partnership    CollegePartnership  @relation(fields: [partnershipId], references: [id], onDelete: Cascade)
  tpoId          String
  tpo            TPO                 @relation(fields: [tpoId], references: [id], onDelete: Cascade)
  studentId      String
  student        Student             @relation(fields: [studentId], references: [id], onDelete: Cascade)
  recruiterId    String
  recruiter      Recruiter           @relation(fields: [recruiterId], references: [id], onDelete: Cascade)
  message        String?
  createdAt      DateTime            @default(now())

  @@index([recruiterId, createdAt])
  @@index([studentId])
  @@map("direct_talent_pushes")
}
```

Add the back-relation arrays (`partnerships CollegePartnership[]`, `directPushes
DirectTalentPush[]`, etc.) to the existing `College`, `Recruiter`, `TPO`, and `Student`
models wherever Prisma requires the inverse side — check the exact existing field names
in those models first so you don't collide with something already there.

**Constraint to enforce in service code, not just schema**: `DirectTalentPush` should
only be allowed to/from a partnership with `status: ACTIVE` — pushing a student to a
recruiter you haven't partnered with yet (or whose request is still PENDING/was REJECTED)
must be rejected with a `ForbiddenException`. Also enforce the pushed student actually
belongs to the TPO's own college (reuse the existing `assertOwnCollegeStudent` pattern
already in `backend/src/modules/tpo/tpo.service.ts`).

## 2. Backend (`backend/src/modules/partnerships/`)

New module, follow the existing module structure/conventions used by
`backend/src/modules/disputes/` (controller + service + dto folder, registered in
`app.module.ts`).

- `POST /partnerships/request/:recruiterId` — caller can be TPO or RECRUITER (check
  `@CurrentUser().role` inside the service, not two separate endpoints) and creates a
  `PENDING` `CollegePartnership` row with `initiatedBy` set to whichever role made the
  call. If a TPO calls this, resolve their own `collegeId` from their TPO record; if a
  RECRUITER calls this, the `:recruiterId` param is them confirming who they want to
  reach — actually, re-think the route shape: a TPO partnering with a recruiter needs to
  specify *which recruiter*; a recruiter partnering with a college needs to specify
  *which college*. Use **one shared endpoint** `POST /partnerships/request` with body
  `{ collegeId?: string, recruiterId?: string }` where the caller supplies the *other*
  party's id (their own side is inferred from their JWT/role) — that avoids needing two
  near-duplicate routes. Return `BadRequestException` if a partnership between that pair
  already exists (any status) rather than creating duplicates — surface the existing
  record's status instead so the frontend can show "Already partnered" / "Request
  pending" appropriately.
- `PATCH /partnerships/:id/accept` — the *other* party (not the initiator) accepts;
  sets `status: ACTIVE`, `respondedAt: now()`. Reject if caller is the initiator
  (can't accept your own request) or if not the correct recruiter/college side.
- `PATCH /partnerships/:id/reject` — same shape, sets `status: REJECTED`.
- `GET /partnerships/my` — returns all partnerships (any status) where the caller's
  college (if TPO) or recruiter profile (if RECRUITER) is a party. Include the other
  side's display info (college name, or recruiter companyName + user name) so the
  frontend doesn't need a second round-trip.
- `POST /tpo/students/:studentId/push-to-recruiter/:recruiterId` — add this to the
  **existing** `backend/src/modules/tpo/tpo.controller.ts` / `tpo.service.ts` (it's a
  TPO action on a student, belongs with the other TPO student actions, not in the new
  partnerships module). Body: `{ message?: string }`. Validates: caller is TPO, student
  belongs to TPO's college, an ACTIVE partnership exists between TPO's college and that
  recruiterId. Creates the `DirectTalentPush` row and sends a notification to the
  recruiter only (per decision #2 above — do NOT notify the student). Follow the
  `notificationsService.send(...)` pattern already used elsewhere in this file/module.
- `GET /tpo/partnered-talent-pushes` or similar on the **recruiter** side (check existing
  recruiter routes in `backend/src/modules/recruiters/recruiters.controller.ts` for where
  this fits) — lists `DirectTalentPush` rows where `recruiterId` = caller, with student
  profile info included (reuse the student-profile shape already used in
  `recruiters.service.ts`'s discover/search method for consistency), most recent first.
